# Production Deployment Design

**Date:** 2026-06-03
**Goal:** Ship the ITE Calibration web application to a Windows PC running Docker Desktop, migrating historical calibration data from both the live Docker stack and the legacy Excel/CSV workflow.

---

## Context

The application is a monorepo:

- `apps/api/` ‚Äî FastAPI (Python 3.12)
- `apps/web/` ‚Äî Vite + React + TypeScript
- `infra/` ‚Äî Docker Compose stack (postgres + api + web + edge nginx)
- `Old Method/` ‚Äî legacy pandas/openpyxl scripts and historical Excel workbooks

A prior production-readiness audit identified 43 gaps; recent commits (`6d75ce8`, `dd90b15`, `38fb1e2`, `8f4e5be`) addressed critical, high/medium, low-priority, and UX items. This spec covers the remaining work to actually deploy.

**State of historical data:**

- The live Docker stack has been running for ~7 days and contains real calibration runs entered through the web UI.
- The legacy `Old Method/` directory contains:
  - 2 multi-sheet workbooks covering 215 loggers (cert serials `190124110002450`‚Äì`190124110002664`).
  - At least 3 distinct historical batches with different starting cert numbers and dates (Batch 1 starts at cert `1645`, test date `2026Âπ¥3Êúà4Êó•`; later batch starts at `1720`; Batch 3 starts at `1935`, test date `2026Âπ¥4Êúà14Êó•`).

The user confirmed reference CSVs are available for every batch.

## Goal

Produce a self-contained `deploy-package/` directory that, when copied to the Windows PC and run with `setup-windows.ps1`, yields a working application with **all** historical calibration data (legacy Excel batches + new runs from the live Docker stack) visible in the History page with proper per-setpoint deviation data.

## Section 1 ‚Äî Code Fixes

**Single change:** Sync `deploy-package/docker-compose.yml` with `infra/docker-compose.prod.yml` so the deploy package includes:

- API container `healthcheck` block (curl `/api/health` every 15s, 5s timeout, 3 retries, 20s start period)
- API container `logging` block (json-file driver, 10MB max-size, 3 max-file)
- Web container `logging` block (json-file driver, 5MB max-size, 3 max-file)

The cleanest way is to copy `infra/docker-compose.prod.yml` over the deploy-package version.

**Why it matters:** Without the healthcheck, Docker can't gate the edge nginx on API readiness ‚Äî nginx may briefly serve 502s after restart. Without log rotation, container logs grow unbounded on the Windows PC.

## Section 2 ‚Äî Historical Data Migration

### Architecture

A standalone Python script `scripts/migrate_historical.py` that:

1. Loads a config file `scripts/historical_batches.json` describing every historical batch.
2. Connects directly to the live Postgres via `ITE_DATABASE_URL` (the same one the API uses).
3. For each batch, for each logger sheet in the workbook, calls the existing calibration engine (`ite_api.calibration.engine.run_one_logger`) to compute per-setpoint reference/calibration values.
4. Writes the certificate `.docx` to the data volume (mounted at `/var/lib/ite-calibration/data` in the API container, accessed from the host through `docker cp` or volume mount).
5. Inserts proper database rows: one `CalibrationRun` (status=`completed`), one `Logger` per sheet, one `LoggerResult` per setpoint, one `RunReferenceFile` and `RunCalibrationFile` per source file, and an `AuditLog` entry marking the run as `migrated`.
6. Is **idempotent**: if a logger serial already exists in the DB it is skipped, so the script can be re-run safely after a partial failure.

### Config schema (`historical_batches.json`)

```json
{
  "batches": [
    {
      "name": "Batch 1 ‚Äî March 2026",
      "workbook": "Old Method/No. 2450 - No. 2554.xlsx",
      "references": ["Old Method/Standard Logger 1.csv", "Old Method/Standard logger 2.csv"],
      "start_cert_no": "0000001645",
      "cert_width": 10,
      "test_date_jp": "2026Âπ¥3Êúà4Êó•",
      "doc_date_jp": "2026Âπ¥3Êúà6Êó•",
      "window_start": "2026-03-04T00:00:00",
      "window_end": "2026-03-04T23:59:59"
    }
  ]
}
```

The user fills this in once based on the constants in the legacy `calibration.py` / `generate_certificates.py` / `generate_certificates_new.py` scripts.

### Why direct engine invocation (not the HTTP API)

The API's `POST /api/runs` flow is designed for the interactive new-calibration wizard ‚Äî it walks a run through `draft` ‚Üí `uploaded` ‚Üí `processing` ‚Üí `completed` with file uploads, background tasks, and 60s polling. Re-using that for 215 historical loggers across 3 batches would be slow and harder to error-handle. Calling `run_one_logger` directly is the same code path the background task uses; we just skip the HTTP/state machine layer and insert the terminal rows ourselves.

### Distinguishing historical from new runs

Each migrated `CalibrationRun` row gets `notes = "migrated from legacy Excel workflow"` and an `AuditLog` entry with `action=run.migrated`. This means the History page shows them as normal completed runs (which is what the user wants), but the audit trail keeps them traceable.

### Execution

The script runs **inside the api container** via `docker exec`. This avoids exposing Postgres to the host network and gives the script direct write access to the mounted `cal_data` volume at `/var/lib/ite-calibration/data`.

```bash
# 1. Copy the script and config into the api container
docker cp scripts/migrate_historical.py ite-calibration-api-1:/app/scripts/
docker cp scripts/historical_batches.json ite-calibration-api-1:/app/scripts/

# 2. Copy the legacy Excel/CSV inputs into the container (read-only mount avoids the copy,
#    but Docker Desktop on macOS makes ad-hoc bind mounts on a running container awkward ‚Äî
#    a one-shot copy is simpler)
docker cp "Old Method" ite-calibration-api-1:/tmp/old_method

# 3. Run the migration ‚Äî uses the same ITE_DATABASE_URL the API already has
docker exec -w /app ite-calibration-api-1 \
  python -m scripts.migrate_historical --config scripts/historical_batches.json \
                                       --inputs-root /tmp/old_method
```

The script writes certificates directly into `/var/lib/ite-calibration/data` (the mounted `cal_data` volume), so no extra copy step is needed.

## Section 3 ‚Äî Build & Package

### Steps

1. **Build images**
   ```bash
   docker build -t ite-calibration-api:latest -f apps/api/Dockerfile apps/api
   docker build -t ite-calibration-web:latest -f apps/web/Dockerfile apps/web
   ```

2. **Save images**
   ```bash
   docker save ite-calibration-api:latest | gzip > deploy-package/images/api.tar.gz
   docker save ite-calibration-web:latest | gzip > deploy-package/images/web.tar.gz
   ```

3. **Dump database** (post-migration so historical runs are included)
   ```bash
   docker exec ite-calibration-postgres-1 \
     pg_dump -U ite --clean --if-exists ite > deploy-package/seed/db_full.sql
   ```

4. **Backup cal_data volume**
   ```bash
   docker run --rm \
     -v ite-calibration_cal_data:/source:ro \
     -v "$(pwd)/deploy-package":/backup \
     alpine tar czf /backup/cal_data.tar.gz -C /source .
   ```

5. **Sync infra files into deploy-package**
   - `infra/docker-compose.prod.yml` ‚Üí `deploy-package/docker-compose.yml`
   - `infra/nginx.conf` ‚Üí `deploy-package/nginx.conf` (already in sync, verify)
   - `infra/setup-windows.ps1` ‚Üí `deploy-package/setup-windows.ps1` (already in sync, verify)
   - `infra/backup-windows.ps1` ‚Üí `deploy-package/backup-windows.ps1` (already in sync, verify)
   - Verify `deploy-package/.env.example` is present

## Section 4 ‚Äî Verify

Simulate the Windows install on this Mac before shipping:

1. **Stop the live stack** (preserves volumes by default):
   ```bash
   cd infra && docker compose down
   ```

2. **Snapshot the live volumes** to a safe location (rollback insurance):
   ```bash
   docker run --rm -v ite-calibration_pg_data:/v -v "$(pwd)/.snapshot":/b \
     alpine tar czf /b/pg_data.tar.gz -C /v .
   ```

3. **Spin up a fresh test environment** in a sibling directory using only the deploy-package contents (different compose project name to avoid clobbering the snapshots):
   ```bash
   cd /tmp && cp -r ~/Projects/Calibration/deploy-package ./verify
   cd verify && cp .env.example .env  # edit secrets
   docker compose -p ite-verify up -d
   # ‚Ä¶ wait for healthcheck, run smoke tests ‚Ä¶
   docker compose -p ite-verify down -v
   ```

### Verification checklist

- [ ] `setup-windows.ps1` (or its bash-equivalent steps) completes without errors.
- [ ] `curl http://localhost/api/health` returns `{"status":"ok",...}`.
- [ ] Login page loads at `http://localhost/`.
- [ ] Login with the seeded admin user succeeds.
- [ ] Overview page shows non-zero stats reflecting historical + new runs.
- [ ] History page lists historical runs (status `completed`, marked with migration notes) **and** newer Docker-stack runs.
- [ ] Clicking a historical run opens Run Detail with populated per-setpoint deviation cells.
- [ ] Downloading a certificate `.docx` from a historical run works and produces a valid document.
- [ ] Downloading "all certificates (.zip)" from a historical run works.

If any check fails, the bug gets fixed and we re-run section 3 + 4. We do **not** ship a partial package.

## Out of Scope

- HTTPS / TLS certificates. The Windows PC deployment is on a closed internal LAN at HTTP only, consistent with the current `cookie_secure=False` setting in `apps/api/ite_api/config.py`. Adding HTTPS would require either a reverse proxy in front of nginx or self-signed certs; deferred.
- Alembic-based upgrade path. The initial install uses the DB seed dump; future schema upgrades will need a documented `alembic upgrade head` step on the Windows PC. Documenting this is a follow-up.
- Importing the legacy PDF certificates. Only the source `.docx` files (regenerated by the new engine) ship in the cal_data volume; the legacy PDFs in `Old Method/output/` stay where they are.
- Automated tests for `scripts/migrate_historical.py`. The script runs once against real data; we verify by inspecting the resulting History page and a spot-checked Run Detail, not by adding pytest coverage.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Reference CSV doesn't match the date window of a historical batch ‚Üí engine returns no match for some setpoints | Use a wide `window_start`/`window_end` per batch; surface "no match" loggers in the script's stdout for manual review |
| Cert number collisions between historical batches and new Docker-stack runs | The migration runs first; new runs in the Docker stack have cert nos starting from a much higher number; verify ranges don't overlap before running |
| Idempotency bug ‚Üí re-running migrates everything twice | Skip-by-serial check is implemented in step 1 of every logger insert; covered by inspecting `Logger.serial` unique constraint |
| `cal_data` volume snapshot taken before migration completes ‚Üí ships incomplete certificates | Section 3 step 4 explicitly runs *after* migration; verify timestamps before packaging |
| Windows PC missing Docker Desktop or PowerShell execution policy blocks the script | `setup-windows.ps1` already preflights Docker; document the `Set-ExecutionPolicy Bypass` flag in deployment notes |

## Implementation Sequence

The implementation plan (next step, via `writing-plans`) will sequence work as:

1. Section 1 ‚Äî sync compose files (smallest change first)
2. Build the migration script + its config (Section 2)
3. User reviews + fills in `historical_batches.json`
4. Run the migration against the live Docker stack
5. Verify migrated data in the UI before packaging
6. Section 3 ‚Äî build images, dump DB, backup volume, assemble package
7. Section 4 ‚Äî verify package on this Mac
8. Ship to the Windows PC
